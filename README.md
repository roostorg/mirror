# mirror

Experiment in mirroring ROOST repos to alternate git forges

## How it works

This repo is powered by a simple [GitHub Actions workflow](.github/workflows/mirror.yml). It works by:

1. Creating a job for every repository included in the `repo` array
2. Checking out the repo and all of its history
3. Adding a remote pointing to the mirror, using a repository secret as the access token
4. Force-pushing all branches and tags to that mirror remote, explicitly pruning deleted branches

The workflow runs automatically every hour, or manually on dispatch (running the workflow from the Actions tab).

### To mirror a repository:

1. Create an empty repository of the same name on the mirror (i.e. at [gitlab.com/roostorg](https://gitlab.com/roostorg/))
2. Add the repository name to the `repo` array in [.github/workflows/mirror.yml](.github/workflows/mirror.yml)
3. Wait up to an hour after merged, or manually run the workflow from the Actions tab 

## Important!

- ROOST org repositories are not mirrored automatically; they must be explicitly added one-by-one

- The workflow does not currently validate that repos are public; **do not mirror private repos**!

- Access tokens must be generated by someone with appropriate permissions on the mirrored account, and may expire (i.e. GitLab expires tokens annually); the workflow will fail if using an expired token

- Only the git repository (branches, tags, and commits) are mirrored; pull requests and issues are not

- **Mirrors should be treated as read-only**; there is no two-way syncronization
